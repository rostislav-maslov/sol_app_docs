# Project Rules

Эта директория хранит project-specific архитектурные и предметные правила
конкретного репозитория.

## Зачем нужна эта директория

Не все правила должны жить в общих `ai-rules`.
То, что относится именно к данному проекту, нужно фиксировать отдельно и
переиспользовать между задачами.

## Что сюда складывать

- правила структуры слоев и пакетов;
- стандарты API;
- правила отчетных endpoint-ов;
- device/domain/entity conventions;
- naming, wiring и boundary-правила;
- любые долгоживущие решения, которые должны применяться повторно.

## Принцип

- если правило привязано к конкретной архитектуре проекта, оно должно жить
  здесь;
- если правило межпроектное и повторяется везде, оно должно жить в
  `ai-context/ai-rules`;
- задачи из `tasks/task-details` должны ссылаться на применимые файлы из этой
  директории.

## Базовая структура по технологическим контурам

Чтобы project-specific правила не смешивались в одном плоском списке, базовый
каркас уже разделен по направлениям:

- `backend/` - правила для backend-контуров и серверной архитектуры;
- `frontend-react-js-ts/` - правила для frontend на `React`, `JavaScript` и
  `TypeScript`;
- `flutter/` - правила для `Flutter` и мобильного клиента.

Пока в этих директориях лежат только `README.md`, а сами стандарты можно
добавлять позже по мере уточнения подхода.

## Именование

Рекомендуется хранить одно устойчивое правило в одном файле.

Примеры имен:

- `01-domain-layer-structure-rules.md`
- `03-api-rules.md`
- `04-dashboard-report-rules.md`

Для нового project-specific правила можно использовать шаблон
`ai-context/rules/_template.md`.
