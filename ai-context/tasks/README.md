# Tasks Context

Пакет `ai-context/tasks` - это единая точка работы с задачами в диалоге с AI.
Он превращает разрозненные запросы в управляемый task-flow.

## Состав пакета

- `task-list.md` - основной короткий список задач.
- `task-details/` - плоские markdown-файлы детализации задач по их коду.
- `task-draft.txt` - сырой или исторический список задач, который можно использовать
  как источник перед форматированием.

Этот `README.md` является единым source-of-truth для workflow и execution-правил
директории `ai-context/tasks`.

## Что такое `task-details`

`task-details` - это не заметки, а готовая постановка для AI.

`task-details` устроен как плоский набор файлов по коду задачи:

```text
task-details/
  _template.md
  <код>.md
```

Где:

- `task-details/_template.md` - baseline-шаблон для новых task detail-файлов;
- `task-details/<код>.md` - основной файл детализации задачи.

Каждая задача из `task-list.md` при детализации должна получать отдельный файл
по своему коду прямо внутри `task-details/`.

Если задача крупная, ее декомпозиция, контекст, ограничения и связанные файлы
описываются в том же `task-details/<код>.md`. Поддиректории для `subtasks/`,
`context/` и иной вложенной структуры больше не используются.

## Код задачи

По умолчанию используется локальный формат `01-001`, но это не единственный
допустимый вариант.

Допустимые варианты кода задачи:

- локальный код репозитория, например `01-001`;
- ключ задачи из Jira, если локальные стандарты репозитория требуют
  использовать внешний код, например `PROJ-123`;
- идентификатор задачи из Bitrix, если локальные стандарты репозитория требуют
  использовать его как основной код, например `B24-12345` или другой
  согласованный формат.

Для локального формата `01-001`:

- `01` - номер списка или потока задач;
- `001` - порядковый номер внутри списка.

Общие правила кода задачи:

- код присваивается задаче один раз и дальше не меняется;
- если в репозитории нет специальных локальных стандартов, новые задачи
  получают следующий свободный локальный код;
- если в репозитории есть локальные стандарты по использованию Jira или
  Bitrix-кодов, нужно следовать им;
- если используется внешний код из Jira или Bitrix, он должен быть стабильным
  и читаемым для человека;
- файл детализации создается как `task-details/<код>.md`.

Если локальные стандарты репозитория не определены, использовать локальный
формат `01-001` как безопасный вариант по умолчанию.

## Карта статусов

- `🔴` - новая задача. Есть только краткая формулировка.
- `🟠` - задача детализирована.
- `🟡` - задача в работе.
- `🟣` - AI завершил работу и ждет ручной проверки пользователя.
- `🟢` - пользователь вручную проверил и подтвердил корректность результата.

Важно:

- AI не должен переводить задачу в `🟢`.
- После завершения работы AI переводит задачу в `🟣`.
- В `🟢` задачу переводит только пользователь.

## Команда `отформатируй задачи`

Если пользователь просит `отформатируй задачи`, нужно:

1. взять неотформатированные строки из `task-list.md` или из указанного сырого
   источника;
2. сохранить порядок задач;
3. добавить код всем задачам, у которых его нет;
4. добавить статус всем задачам, у которых его нет;
5. использовать статус `🔴` по умолчанию;
6. привести каждую строку к формату `🔴 [<код>] Короткое описание задачи`.

Если локальные стандарты репозитория требуют использовать Jira key или Bitrix
task id как код задачи, при форматировании нужно использовать именно этот
формат, а не принудительно переводить все в локальный `01-001`.

## Команда `детализируй <код>`

Если пользователь просит `детализируй <код>`, нужно:

1. найти задачу по коду в `task-list.md`;
2. создать файл `task-details/<код>.md`;
3. заполнить его по шаблону `task-details/_template.md`;
4. обновить статус задачи в `task-list.md` на `🟠`, если работа еще не началась.

При детализации нужно:

- описать цель и ожидаемый результат;
- перечислить ограничения и связанные файлы;
- явно зафиксировать обязательные дополнительные условия;
- сослаться на применимые `ai-context/rules/*.md`, если они есть;
- если задача крупная, описать декомпозицию прямо внутри `task-details/<код>.md`
  без создания поддиректорий.

## Execution-правила

### Правило статусов задач

#### Цель

Зафиксировать единый стандарт: AI отмечает факт завершения своей работы, а
пользователь отдельно подтверждает итог после ручной проверки.

#### Триггер

Правило применяется ко всем задачам, которые ведутся через
`ai-context/tasks/task-list.md`.

#### Обязательная карта статусов

- `🔴` - новая задача без детализации.
- `🟠` - задача детализирована, для нее создан файл в `task-details/`.
- `🟡` - задача в работе.
- `🟣` - AI завершил работу и ожидает пользовательской проверки.
- `🟢` - пользователь сам проверил результат и подтвердил корректность.

#### Обязательное поведение AI

- После детализации переводить задачу из `🔴` в `🟠`.
- Когда задача берется в работу, допустимо ставить `🟡`.
- После завершения реализации, документации или иной фактической работы AI
  должен ставить `🟣`, а не `🟢`.
- AI не должен самостоятельно переводить задачу в `🟢`, даже если локальные
  проверки прошли успешно.
- `🟢` ставит только пользователь после собственной проверки.

#### Допустимые переходы

- `🔴 -> 🟠`
- `🟠 -> 🟡`
- `🟡 -> 🟣`
- `🟠 -> 🟣`, если задача выполнена без отдельной фиксации `🟡`
- `🔴 -> 🟣`, если пользователь просит сразу выполнить маленькую задачу
- `🟣 -> 🟢` только после ручной проверки пользователем

### Правило алерта о завершении задачи

#### Цель

После завершения любой задачи нужно показать модальный алерт, который нельзя
пропустить в потоке чата.

#### Триггер

Правило применяется:

- после завершения любой задачи;
- если пользователь отдельно просит показать уведомление по завершении.

#### Обязательный способ

На macOS использовать только локальный скрипт:

```bash
./ai-context/scripts/show-completion-alert.sh "Codex" "Работа завершена."
```

#### Что запрещено как основной способ

- не считать текст в чате заменой модальному окну;
- не использовать `echo`, `printf`, `\a` и подобные способы как единственный
  сигнал завершения;
- не вызывать `osascript` напрямую, если в репозитории уже есть baseline-скрипт.

#### Подтверждение выполнения

В финальном ответе нужно кратко указать, что алерт был показан, либо явно
сообщить, что среда не позволила его показать.

### Стандарт резюме для коммита

#### Цель

После кодовых и документационных изменений финальный ответ должен содержать
готовое резюме, которое можно использовать как commit message и commit body.

#### Триггер

Правило применяется:

- если в рамках задачи были внесены изменения в файлы;
- если пользователь просит подготовить текст для коммита.

#### Обязательный формат

В конце финального ответа добавить секцию `Резюме для коммита` и fenced `text`
блок в формате:

```text
<type>(<scope>): <краткий заголовок на русском> #ai

- <ключевое изменение 1>
- <ключевое изменение 2>
- <ключевое изменение 3>
```

Где:

- `<type>`: `feat`, `fix`, `docs`, `chore`, `refactor`, `test`, `data`,
  `rules`;
- `<scope>`: область изменений, например `ai-context`, `tasks`, `api`,
  `domain`;
- заголовок: одна строка, конкретно про результат;
- список: только фактические изменения без общих фраз.

Если изменений в файлах не было, в блоке нужно написать:

```text
коммит не требуется
```

#### Что запрещено

- размытое описание без конкретных изменений;
- смешивание в одном summary нерелевантных задач;
- отсутствие хештега `#ai` в заголовке коммита при реальных изменениях;
- пропуск секции `Резюме для коммита` после file changes.

### Правило changelog

#### Цель

После любых file changes оставлять запись в `ai-context/changelog`, чтобы
история работы AI сохранялась внутри репозитория.

#### Триггер

Правило применяется, если в рамках задачи были изменены, добавлены или удалены
файлы проекта.

#### Обязательный формат changelog

- использовать директорию `ai-context/changelog`;
- создавать отдельный changelog-файл на каждое завершенное изменение;
- использовать имя файла в формате `YYYY-MM-DD_HHMMSS_short-title.md`;
- после создания не дописывать в этот файл другие изменения;
- каждый changelog-файл должен содержать:
  - время;
  - короткий заголовок;
  - список измененных файлов или областей;
  - краткое описание фактически выполненных изменений.

Пример:

```md
# Переход на one-change-one-changelog

- Время: `2026-03-24 15:47:59`
- Файлы: `ai-context/ai-rules/005_append-only-changelog.md`, `ai-context/changelog/README.md`
- Что сделано: правило changelog переведено на модель «одно изменение = один файл», чтобы снизить git-конфликты.
```

Исторические changelog-файлы, созданные по старой дневной схеме, можно
сохранять как есть.

#### Что запрещено

- пропускать запись в changelog после реальных file changes;
- дописывать новые изменения в уже существующий changelog-файл;
- перезаписывать старые changelog-файлы вместо создания нового;
- писать абстрактные записи без перечисления фактических изменений.
