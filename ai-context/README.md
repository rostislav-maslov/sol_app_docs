# AI Context

`ai-context` - это переносимый baseline-пакет для рабочих репозиториев, в
котором AI-агенту заранее даны:

- фиксируемый контекст диалога;
- рабочий backlog и task-flow;
- append-only changelog;
- постоянные правила работы агента;
- место для project-specific архитектурных правил;
- служебные скрипты и переиспользуемые промты.

Структура и рабочая модель собраны по образцу зрелого `ai-context` из
`personal_ai_agent_backend`, а компактные постоянные правила и служебные
скрипты выровнены по подходу `ceo-ai`.

## Состав

- `ai-rules/` - постоянные межпроектные правила поведения AI-агента.
- `rules/` - project-specific архитектурные и предметные правила конкретного
  репозитория.
- `parameters/` - параметры использования `ai-context` в двух scope:
  repository-level и local-machine-level.
- `tasks/` - рабочий task-flow: список задач, детализация и execution-правила.
- `changelog/` - append-only журнал изменений по принципу «одно изменение =
  один файл».
- `content/` - проектные markdown-артефакты контекста и рабочие материалы.
- `scripts/` - локальные утилиты, которые агент должен вызывать единым способом.
- `promts/` - переиспользуемые промты для типовых операций.

Имя директории `promts` оставлено в текущем виде для совместимости с уже
существующими репозиториями.

## Разделение AI-задач и эпиков команды

`ai-context/tasks/` остается контуром задач, которые должен выполнить сам
AI-агент.

Если репозиторий работает в режиме `project-manager`, задачи и эпики для
команды не должны смешиваться с `ai-context/tasks/`. Для них используется
отдельная корневая директория `epics/`, где:

- `epics/epic-list.md` - короткий список эпиков;
- `epics/<код>/<код>-epic.md` - основная постановка эпика;
- дополнительные `<код>.md`, `context/` и иные материалы живут рядом внутри
  директории эпика.

Важно:

- в этом source-of-truth репозитории `epics/` хранится как документационный
  baseline-шаблон;
- в реальный проект `epics/` копируется только в режиме `project-manager`;
- из baseline переносятся только документационные файлы, а живые `epic-list`
  и директории эпиков остаются project-local данными целевого проекта.

## Рабочий цикл

1. Определи контур задачи:
   - задача для AI - ведется через `ai-context/tasks/task-list.md`;
   - эпик или задача для команды в режиме `project-manager` - ведется через
     корневую директорию `epics/`.
2. Для AI-задачи при необходимости создай или обнови файл детализации
   `ai-context/tasks/task-details/<код>.md`.
3. Для эпика команды при необходимости создай или обнови
   `epics/<код>/<код>-epic.md`, `epics/<код>/<код>.md` и supporting-материалы.
4. Перед реализацией проверь применимые `ai-context/ai-rules/*.md`,
   `ai-context/tasks/README.md` и project-specific `ai-context/rules/*.md`.
5. После изменений переведи AI-задачу в `🟣`, обнови `ai-context/changelog`,
   покажи модальный алерт и подготовь резюме для коммита.
6. Только пользователь после ручной проверки переводит задачу в `🟢`.

## Установка и обновление

Текущий source-of-truth для baseline-части `ai-context`:

- `https://github.com/foodtechlab/ai_context_rules`

Актуальную версию baseline нужно тянуть из основной ветки репозитория:

- `main`, если она существует;
- `master`, если в репозитории основная ветка называется так.

Критично различать:

- baseline-файлы, которые можно копировать и обновлять из шаблона;
- project-local файлы, которые нельзя затирать при синхронизации.

Подробная политика описана в `ai-context/update-policy.md`.

Если агент устанавливает или обновляет `ai-context` из этого репозитория в
рабочий проект, он должен сначала прочитать `ai-context/update-policy.md`, а
уже потом копировать файлы.
